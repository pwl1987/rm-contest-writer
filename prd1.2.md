## 四、python-docx 优化方向

### 4.1 推荐依赖升级

当前项目使用 `python-docx` 的最新稳定版为 **1.2.0**（2025-06-17 发布），建议在 `requirements.txt` 中锁定：

```
python-docx>=1.1.0,<2.0.0
lxml>=4.9.0
```

python-docx 1.2.0 新增了对样式对象的完整支持，可以进一步优化公文模板中的样式管理。

### 4.2 公文格式完整性优化

当前脚本已适配 GB/T 9704-2012 的核心要求。基于对同类方案的研究，以下 3 项可进一步增强：

**（1）页码格式修正**

当前页码仅居中显示数字（L140–148）。GB/T 9704-2012 要求页码为 4 号半角宋体阿拉伯数字，左右各放一条一字线（`— 5 —`）。建议修改 `set_page_headers` 中的页码生成逻辑：

```python
# 页码格式：一字线 + 空格 + 页码 + 空格 + 一字线
run.text = '— '
# 插入 PAGE 域代码
run.text += ' —'
```

**（2）版头红线（分割线）**

正式公文首页版头（单位名称下方）需有一条红色分隔线（武文线）。建议在 `add_unit_name_paragraph` 后新增 `add_header_separator(doc)` 方法。

**（3）联合行文署名格式**

当前 `is_signature_line` 仅识别单一署名格式。建议扩展对联合行文（多个单位并列署名）的支持。

### 4.3 截图嵌入流程优化

当前 `capture-diagram.py`（48 行）优先调用 Node.js 版本，Python playwright 仅为备用。如果目标是**纯 Python 实现**，建议：

- 将 Python playwright 方案升为**主方案**，移除对 Node.js 的硬依赖
- 或保留双方案但增加**优雅降级提示**（如“Node.js 不可用，已自动切换 Python playwright”）

```python
# 当前逻辑
if capture_with_node(...):
    return
if check_playwright_python():
    capture_with_python(...)
# 问题：两个方案都失败时，静默退出，无错误提示

# 建议改为
# 方案1：尝试 Playwright（Python）
# 方案2：降级为纯文本架构描述 + 手动截图提示
```

### 4.4 字体依赖声明

当前 SKILL.md 和脚本中均未明确声明字体文件的来源与许可。`fonts/` 目录中的方正字体涉及商业授权，建议在 README 中添加：

> ⚠️ 本项目使用方正字体仅用于符合 GB/T 9704-2012 的公文格式展示。商用需获取方正字库授权。
