# 质量评分卡（Gate 校验体系）

> **评审者角色：严苛的甲方评审**
> - 禁止自我辩护：不得以"考虑到..."、"虽然..."等句式为低分项辩护
> - 必须引用标准：评分时必须注明依据的制度条款（如 GB/T 9704-2012 第 5.1 条）
> - 强制挑刺原则：每项评分必须给出至少一条可改进的具体意见（T:8 以下必填）
> - 零容忍模糊表述：禁止"基本符合"、"大体合理"等模糊措辞

> **执行规则**：任意 [BLOCK] 触发 → **立即停止文档生成**（不输出正文，不进入第六步），输出修改建议并返回第三步重新撰写。

> **重要**：以下 5 个 [BLOCK] Gate 为强制性校验。**任何 Gate 不通过，流程必须停止**，不允许跳过或降级处理。

---

## Gate 1：实体一致性校验

成稿后，**逐项**核查以下实体引用，必须完全匹配：

| 检查项 | 要求 | 当前值（从文中提取） | 一致？ |
|-------|------|-------------------|--------|
| model_choice | 3.3节关键技术中的模型名称必须与 model_choice 完全一致 | | □是 □否 |
| GPU型号 | 3.3节 GPU 型号、数量与 5.2节 资源需求完全一致 | | □是 □否 |
| 测试环境硬件 | 3.3节 测试环境（A100/RTX 4090）与 5.2节 部署环境一致 | | □是 □否 |
| 软件授权 | 3.3节 开源模型 vs 商业 API 与 5.2节 软件授权费对应 | | □是 □否 |

**[BLOCK]** 若任何一项为「否」：立即停止文档生成，输出修改建议并返回第三步重新撰写。

**常见错误模式：**
- 3.3节写"GLM-4"（纯文本），5.2节部署环境写"GPU服务器" → 矛盾，GLM-4 是 API 调用无需 GPU
- 3.3节写"测试用 A100×1"，5.2节写"部署 RTX 4090×2" → 性能差距 5-10 倍，数据无法复现

## Gate 4：量化对比表数据溯源 [BLOCK]

检查 3.4 节对比表和 4.2 节效果数据：

**3.4 节对比表完整性（技术研发类必检）：**
- [ ] 包含「数据来源」列，注明测试日期、测试环境（硬件配置）
- [ ] 包含准确率行
- [ ] 包含召回率行（recall row，极易遗漏）
- [ ] 包含误报率行

**4.2 节效果数据格式：**
- [ ] 禁止纯百分比（如「准确率提升 30%」），必须包含绝对数值
- [ ] 正确格式：「准确率 98.5%（绝对值），较基线提升 30%」

**[BLOCK]** 若数据来源列缺失，或召回率行缺失，或效果数据为纯百分比：立即停止文档生成，输出修改建议并返回第三步重新撰写。

## Gate 5：core_functions 关键词 exact-match [BLOCK]

从 `memory/projects/{project_id}.json` 中读取 `core_functions` 数组。

在 3.2 节核心功能中，**每个** core_functions 词组必须**完整出现**（不允许同义替换）：

> 示例：若 core_functions = ["信号监测", "故障报警", "自动切换", "延时联动"]
> 则 3.2 节必须出现这四个**完整词组**，不允许：
> - "信号监测" → "信号监控"（同义替换，不允许）
> - "故障报警" → "异常告警"（同义替换，不允许）

**检查方法：** 搜索每个 core_functions 词组是否作为**子字符串**出现在 3.2 节中。

**[BLOCK]** 若任何 core_functions 词组缺失或被替换：立即停止文档生成，输出修改建议并返回第三步重新撰写。

## Gate 6：技术栈时效性 [BLOCK]

检查方案中引用的技术/模型是否为当前可用最新状态。

**模型时效性（技术研发类必检）：**
- [ ] 引用的模型厂商未停服（如 GLM-4 API 仍可用、Qwen2.1-72B 仍可申请）
- [ ] 未使用已停止维护的框架版本（如 TensorFlow 1.x、PyTorch <2.0）
- [ ] 开源项目最近一次提交/发版在近 12 个月内

**技术栈时效性（全部赛道）：**
- [ ] 未使用已明确废弃的技术（如 Flash、Python2、Vue2）
- [ ] 主流技术选型在近 3 年内（如 2023-2026 年的主流 AI 架构）

**[BLOCK]** 若使用已停服/明显过时技术：立即停止，输出修改建议并返回第三步。

## Gate 7：落地可行性 [BLOCK]

检查方案资源需求与实施计划是否匹配实际约束。

| 检查项 | 验证逻辑 | 触发条件 |
|--------|---------|---------|
| 预算合理性 | 预算范围与项目复杂度匹配（参考：技术研发类 10-50万，应用创新类 5-30万） | 预算低于复杂度 50% 以上 |
| 周期合理性 | 实施周期与功能数量匹配（参考：每 2 个核心功能 ≈ 1 个月） | 周期短于最低要求 |
| 团队匹配度 | 团队人数与项目规模匹配（参考：技术研发类 ≥3人，应用创新类 ≥2人） | 团队人数明显不足 |
| 硬件需求 | GPU/服务器配置与 ai_mode 匹配（API模式无需 GPU，私有化必须包含 GPU） | 硬件需求与模式矛盾 |

**[BLOCK]** 若任何一项明显不匹配：立即停止，输出修改建议并返回第三步。

## Gate 8：试点数据验证 [BLOCK]

检查方案中是否包含真实试点/验证数据。

**试点数据要求：**
- [ ] 4.2 节效果数据中，必须包含至少 1 项**已在真实环境验证的数据**
- [ ] 真实试点数据标注来源（如"临沂市融媒体中心试点 2026年4月"）
- [ ] 不能全部为"计划中""预计""预期"等未落地数据

**允许"计划中"的情况：**
- 新建项目（2026年尚未开始实施）可全部为计划数据，但需在 5.2 节明确说明
- 已在试点阶段的项目，必须提供至少 1 项真实试点数据

**[BLOCK]** 若全部为计划数据且项目周期显示早已开始：立即停止，标注缺失试点数据项。
